# HeptaSieve

**الخصوصية أولاً، آمن للذكاء الاصطناعي. أداة محلية لمزامنة Heptabase بشكل مستمر إلى Markdown منظم.**

أنت تقرر بالضبط أي البطاقات يمكن لعميل الذكاء الاصطناعي رؤيتها. كل ما عدا ذلك يبقى خارج متناوله.

## لماذا وُجدت هذه الأداة

بدأ الأمر بهدف بسيط: توصيل Heptabase بـ Claude Code كي يتمكن عميل الذكاء الاصطناعي من قراءة ملاحظاتي.

المسار الرسمي هو [CLI](https://github.com/heptameta/heptabase-cli-skills) الخاص بـ Heptabase، تُفعّله في التطبيق من Settings، ثم AI Features، ثم CLI. وهو **fail-open**: بمجرد المنح، يستطيع العميل قراءة قاعدة المعرفة بأكملها. أدوات الجهات الخارجية مثل خادم `heptabase-mcp` تعمل بالطريقة ذاتها. هذا مقبول إذا كان كل شيء في قاعدة معرفتك آمناً للمشاركة. لكن إذا كانت لديك بطاقات سرية جانب البطاقات التي تريد استخدامها مع الذكاء الاصطناعي، وهو حال معظم الناس، فهذا المسار لا يناسبك.

الإدراك الجوهري: يجب أن يقف جدار الخصوصية عند حدود *ما يستطيع الذكاء الاصطناعي قراءته*. هذا الحد يقع خارج Heptabase، في طريقة تغذية ملاحظاتك للعميل. لذا فإن المهمة الحقيقية لأداة كهذه هي **إبقاء البطاقات السرية في مكان لا يصل إليه الذكاء الاصطناعي، وتصدير البقية فقط كـ Markdown يمكن للذكاء الاصطناعي قراءته.** مزامنة الملاحظات هي النصف السهل.

هذه هي المنخل (sieve). فقط البطاقات التي تسمح بها تعبر.

## ما تفعله الأداة

تقرأ HeptaSieve قاعدة بيانات Heptabase المحلية مباشرةً وتكتب البطاقات المختارة كملفات Markdown في الوجهات التي تحددها. تنفّذ مهمة `launchd` هذا كل 15 دقيقة ليبقى Markdown متزامناً مع ملاحظاتك. لا يقرأ عميل الذكاء الاصطناعي سوى مجلد Markdown المصدَّر. لا يلمس قاعدة البيانات أبداً.

- **تقرأ قاعدة البيانات المحلية المباشرة.** توقفت Heptabase عن تقديم [النسخ الاحتياطية المحلية التلقائية](https://support.heptabase.com/en/articles/11064116-how-does-auto-backup-work-in-heptabase) في أواخر 2025، فأصبح قراءة قاعدة البيانات المباشرة هو المسار الموثوق للمزامنة المستمرة.
- **تحويل أمين للبنية.** الجداول، وقوائم bullet / todo / toggle، والأقسام المتداخلة، ومقاطع الفيديو تُستخرج بهندسة عكسية من مخطط ProseMirror الخاص بـ Heptabase وتُصدَّر كـ Markdown نظيف.
- **توجيه إلى أي وجهة.** يمكن وضع كل لوحة بيضاء في مجلدها الخاص، بما في ذلك مسار مطلق يضع اللوحة مباشرةً في مشروع منفصل.

## نموذج الخصوصية fail-closed

لا تُصدَّر بطاقة إلا إذا طابقت إحدى قائمتَي السماح الصريحتين. لا شيء يُقرأ افتراضياً.

| المصدر | القاعدة |
|---|---|
| **`whitelist_whiteboards`** | اللوحات البيضاء التي تسميها. تُقرأ فقط البطاقات الموجودة على *سطح* كل لوحة. اللوحات الفرعية لا تُتابَع. لتضمين إحداها، سمِّها أيضاً. |
| **`card_map`** | طبقة `العنوان -> المسار الدقيق`. هذه العناوين تُزامَن دائماً ومساراتها لها الأولوية. |
| **`blacklist_whiteboards`** | بطاقات هذه اللوحات تُطرح *قبل* قراءة أي محتوى. القائمة السوداء تتغلب على البيضاء، فبطاقة وُضعت بالخطأ في لوحتين تُحجب كذلك. |
| **اللوحات الفرعية (غير المسماة)** | نقل بطاقة إلى لوحة فرعية يغير `whiteboard_id` الخاص بها، فلا يراها مسح السطح. مستبعدة بالبنية لا بقاعدة يجب تذكّرها. |

الضمان في جملة واحدة: كل استعلام يلمس عنوان البطاقة أو محتواها مقيَّد بمعرفات اللوحات في القائمة البيضاء أو عناوين `card_map`. عنوان البطاقة غير المدرجة ومحتواها لا يُحمَّلان أبداً في الذاكرة.

يترتب على ذلك مبدآن في التصميم. **الاستبعاد البنيوي يتفوق على الاستبعاد الطرحي**: بطاقة لا يصلها الاستعلام أأمن من بطاقة تُصفَّى بعد القراءة. **أفضل إشعار هو الذي لا تحتاجه**: صُممت الأداة كي لا تتساءل أبداً إذا كانت بطاقة قد تسرّبت.

## المقارنة

| | HeptaSieve | CLI Heptabase الرسمي | أدوات التصدير الأخرى |
|---|---|---|---|
| نموذج الخصوصية | قائمة سماح fail-closed | Fail-open (قاعدة المعرفة كاملة) | تصدير كامل |
| مزامنة محلية مستمرة | نعم (`launchd`، 15 دقيقة) | قراءة عند الطلب | تصدير لمرة واحدة |
| تقرأ قاعدة البيانات المباشرة | نعم | يتفاوت | غالباً تحتاج ملف نسخة احتياطية |
| Markdown أمين للبنية | جداول، قوائم، أقسام، فيديو | يتفاوت | يتفاوت |
| توجيه وجهة خاصة لكل لوحة | نعم، بما في ذلك المسارات المطلقة | لا | لا |

ليست هذه بديلاً كاملاً لـ Heptabase، و"أفضل من الرسمي" صحيح في ثلاثة محاور فقط: خصوصية قابلة للتحكم، ومزامنة محلية مستمرة، وأمانة البنية. الجمهور المستهدف ضيق بقصد: مستخدمو macOS الذين يعيشون في Heptabase ويهتمون بما يمكن للذكاء الاصطناعي رؤيته. إن كنت من هؤلاء، فهذه الأداة مبنية بالضبط لحالتك.

## التثبيت

المتطلبات: macOS، Python 3.9+، تطبيق Heptabase المكتبي مثبَّت.

```bash
git clone https://github.com/yyu0310/heptasieve.git
cd heptasieve
cp config.example.json config.json
```

ثم عدّل `config.json` (كل حقل يحتوي تعليقاً توضيحياً مضمناً):

1. تأكد أن `db_path` يشير إلى ملف `hepta.db` المحلي.
2. اضبط `base_output_dir` و`board_output_dir` على المكان الذي تريد كتابة Markdown فيه.
3. أدرج اللوحات البيضاء المراد تصديرها في `whitelist_whiteboards`.
4. أضف تجاوزات المسار الدقيقة في `card_map`.

نفّذ أولاً معاينةً لا تكتب شيئاً:

```bash
python3 heptabase_sync.py --dry
```

حين تبدو الخطة صحيحة، نفّذها فعلياً:

```bash
python3 heptabase_sync.py
```

### مزامنة تلقائية كل 15 دقيقة

```bash
cp com.example.heptasieve.plist ~/Library/LaunchAgents/
# عدّل الملف المنسوخ: اضبط المسارات المطلقة وتأكد من مسار python3
launchctl load ~/Library/LaunchAgents/com.example.heptasieve.plist
```

## الاستخدام مع عميل ذكاء اصطناعي

تأتي HeptaSieve مع وثائق يمكن للعميل قراءتها، لتتمكن من الإعداد بالحديث مع عميل ترميز ذكاء اصطناعي بدلاً من اتباع خطوات يدوية:

- [`AGENTS.md`](AGENTS.md) و[`CLAUDE.md`](CLAUDE.md): كيف ينبغي للعميل فهم هذه الأداة وتكوينها.
- [`llms.txt`](llms.txt): فهرس الوثائق للنماذج اللغوية الكبيرة.
- [`skills/setup-heptasieve/`](skills/setup-heptasieve/): مهارة Claude Code تقود الإعداد بأكمله في طلب واحد.

وجّه عميلك إلى مجلد Markdown المصدَّر، ولا توجّهه أبداً إلى `hepta.db`. هذا الفصل هو جوهر الأداة.

## كيف تعمل

راجع [`ARCHITECTURE.md`](ARCHITECTURE.md) للبنية المعمارية: تدفق البيانات، وترتيب fail-closed داخل `build_plan`، وجداول قاعدة البيانات التي تقرأها، والثوابت الخصوصية التي يجب الحفاظ عليها عند تعديل الكود.

## القيود والتحذيرات الصريحة

- **المخطط هش.** يعتمد على البنية الداخلية لقاعدة بيانات Heptabase. قد يكسره تحديث لـ Heptabase. وهو غير رسمي بطبيعته.
- **قراءة قاعدة البيانات المباشرة ليست معتمدة رسمياً.** تعمل بشكل جيد عملياً وهي للقراءة فقط، لكن يجب أن تعلم أنها ليست تكاملاً مدعوماً.
- **macOS فقط.** المسارات وإعداد `launchd` تفترض macOS حالياً.
