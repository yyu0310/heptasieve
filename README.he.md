# HeptaSieve

**פרטיות קודמת, בטוח לבינה מלאכותית. כלי מקומי לסנכרון רציף של Heptabase ל-Markdown מובנה.**

אתם מחליטים בדיוק אילו כרטיסים סוכן הבינה המלאכותית רשאי לראות. כל השאר נשאר מחוץ להישגו.

## למה זה קיים

הכל התחיל ממטרה פשוטה: לחבר את Heptabase ל-Claude Code כדי שסוכן בינה מלאכותית יוכל לקרוא את הפתקים שלי.

הדרך הרשמית היא [CLI](https://github.com/heptameta/heptabase-cli-skills) של Heptabase עצמה, שמופעל באפליקציה דרך Settings, AI Features, CLI. זה **fail-open**: ברגע שמתירים, הסוכן יכול לקרוא את כל בסיס הידע. כלים של צד שלישי כמו שרת `heptabase-mcp` עובדים באותה דרך. זה בסדר אם כל מה שיש לכם בבסיס הידע בטוח לשיתוף. אבל אם יש לכם כרטיסים סודיים לצד כרטיסים שאתם רוצים שהבינה המלאכותית תשתמש בהם, וזה המצב של רוב האנשים, הדרך הזו לא מתאימה.

ההבנה המרכזית: חומת הפרטיות חייבת לעמוד על הגבול של *מה הבינה המלאכותית יכולה לקרוא*. הגבול הזה נמצא מחוץ ל-Heptabase, בדרך שבה אתם מזינים פתקים לסוכן. לכן המשימה האמיתית של כלי כזה היא **לשמור את הכרטיסים הסודיים במקום שהבינה המלאכותית לא מגיעה אליו, ולייצא רק את השאר כ-Markdown שהבינה המלאכותית יכולה לקרוא.** סנכרון הפתקים הוא החצי הקל.

זה הנפה (sieve). רק הכרטיסים שאתם מאשרים עוברים.

## מה זה עושה

HeptaSieve קוראת את מסד הנתונים המקומי של Heptabase ישירות וכותבת כרטיסים נבחרים כקבצי Markdown ביעדים שתבחרו. משימת `launchd` מריצה אותה כל 15 דקות כדי ש-Markdown יישאר מסונכרן עם הפתקים. סוכן הבינה המלאכותית קורא רק את תיקיית ה-Markdown המיוצאת. הוא לא נוגע במסד הנתונים.

- **קוראת את מסד הנתונים המקומי החי.** Heptabase הפסיקה להציע [גיבויים מקומיים אוטומטיים](https://support.heptabase.com/en/articles/11064116-how-does-auto-backup-work-in-heptabase) בסוף 2025, ולכן קריאת מסד הנתונים החי היא כיום הדרך האמינה לסנכרון מקומי רציף.
- **המרה נאמנה למבנה.** טבלאות, רשימות bullet / todo / toggle, קטעים מקוננים וסרטוני וידאו מפוענחים בהנדסה לאחור מסכמת ProseMirror של Heptabase ומוצגים כ-Markdown נקי.
- **ניתוב לכל יעד.** כל לוח לבן יכול לנחות בתיקיית משלו, כולל נתיב מוחלט שמניח לוח ישירות בפרויקט נפרד.

## מודל הפרטיות fail-closed

כרטיס מיוצא רק אם הוא תואם לאחת משתי רשימות הרשאה מפורשות. כברירת מחדל לא נקרא כלום.

| מקור | כלל |
|---|---|
| **`whitelist_whiteboards`** | לוחות לבנים שאתם מציינים בשם. נקראים רק כרטיסים על *פני השטח* של כל לוח. לוחות משנה לא נעקבים. כדי לכלול אחד, ציינו אותו בשם גם כן. |
| **`card_map`** | שכבת `כותרת -> נתיב מדויק`. כותרות אלה תמיד מסונכרנות והנתיב שלהן גובר. |
| **`blacklist_whiteboards`** | כרטיסים על הלוחות האלה מורחקים *לפני* שנקרא כל תוכן. הרשימה השחורה גוברת על הלבנה, כך שכרטיס שהונח בטעות על שני לוחות עדיין חסום. |
| **לוחות משנה (ללא שם)** | להעביר כרטיס ללוח משנה משנה את `whiteboard_id` שלו, ולכן סריקת פני השטח לא רואה אותו. מוחרג במבנה, לא בכלל שצריך לזכור. |

הערבות בשורה אחת: כל שאילתה שנוגעת בכותרת כרטיס או בתוכנו מוגבלת למזהי לוחות ברשימה הלבנה או לכותרות `card_map`. כותרת ותוכן של כרטיס שאינו ברשימה לא נטענים לזיכרון כלל.

שני עקרונות עיצוב נובעים מכך. **החרגה מבנית עדיפה על החרגה חיסורית**: כרטיס שהשאילתה לא מגיעה אליו בטוח יותר מכרטיס שמסוננן אחרי קריאה. **ההתראה הטובה ביותר היא שאינך זקוק לה**: הכלי בנוי כך שלעולם לא תצטרכו לתהות האם כרטיס דלף.

## השוואה

| | HeptaSieve | CLI רשמי של Heptabase | כלי ייצוא אחרים |
|---|---|---|---|
| מודל פרטיות | רשימת הרשאה fail-closed | Fail-open (כל בסיס הידע) | ייצוא מלא |
| סנכרון מקומי רציף | כן (`launchd`, 15 דקות) | קריאה לפי דרישה | ייצוא חד-פעמי |
| קורא מסד נתונים מקומי חי | כן | משתנה | לרוב דורש קובץ גיבוי |
| Markdown נאמן למבנה | טבלאות, רשימות, קטעים, וידאו | משתנה | משתנה |
| ניתוב יעד נפרד לכל לוח | כן, כולל נתיבים מוחלטים | לא | לא |

זה לא תחליף מלא ל-Heptabase, ו"טוב יותר מהרשמי" נכון רק בשלושה ממדים: פרטיות שניתן לשלוט בה, סנכרון מקומי רציף ונאמנות מבנה. הקהל המיועד צר בכוונה: משתמשי macOS שחיים ב-Heptabase ואכפת להם מה הבינה המלאכותית יכולה לראות. אם זה אתם, הכלי הזה נבנה בדיוק למקרה שלכם.

## התקנה

דרישות: macOS, Python 3.9+, אפליקציית שולחן העבודה של Heptabase מותקנת.

```bash
git clone https://github.com/yyu0310/heptasieve.git
cd heptasieve
cp config.example.json config.json
```

לאחר מכן ערכו את `config.json` (לכל שדה יש הערת inline המסבירה אותו):

1. אשרו ש-`db_path` מצביע על `hepta.db` המקומי שלכם.
2. הגדירו את `base_output_dir` ו-`board_output_dir` למקום שבו אתם רוצים שה-Markdown ייכתב.
3. רשמו את הלוחות שאתם רוצים לייצא ב-`whitelist_whiteboards`.
4. הוסיפו עקיפות נתיב מדויקות ב-`card_map`.

הריצו תחילה תצוגה מקדימה שאינה כותבת כלום:

```bash
python3 heptabase_sync.py --dry
```

כשהתוכנית נכונה, הריצו אותה ממש:

```bash
python3 heptabase_sync.py
```

### סנכרון אוטומטי כל 15 דקות

```bash
cp com.example.heptasieve.plist ~/Library/LaunchAgents/
# ערכו את הקובץ שהועתק: הגדירו נתיבים מוחלטים ואשרו את נתיב python3 שלכם
launchctl load ~/Library/LaunchAgents/com.example.heptasieve.plist
```

## שימוש עם סוכן בינה מלאכותית

HeptaSieve מגיעה עם מסמכים שסוכנים יכולים לקרוא, כך שתוכלו להגדיר אותה בשיחה עם סוכן קידוד בינה מלאכותית במקום לעקוב אחר שלבים ידנית:

- [`AGENTS.md`](AGENTS.md) ו-[`CLAUDE.md`](CLAUDE.md): כיצד סוכן צריך להבין ולהגדיר כלי זה.
- [`llms.txt`](llms.txt): אינדקס מסמכים למודלים לשוניים גדולים.
- [`skills/setup-heptasieve/`](skills/setup-heptasieve/): מיומנות Claude Code שמנחה את כל ההגדרה בבקשה אחת.

כוונו את הסוכן שלכם לתיקיית ה-Markdown המיוצאת, לעולם לא ל-`hepta.db`. ההפרדה הזו היא כל הנקודה.

- **הסכמה שבירה.** זה תלוי במבנה מסד הנתונים הפנימי של Heptabase. עדכון Heptabase עלול לשבור אותו. זה לא רשמי מטבעו.
- **קריאת מסד הנתונים החי אינה מאושרת רשמית.** זה עובד היטב בפועל וזה לקריאה בלבד, אבל כדאי לדעת שזו לא אינטגרציה נתמכת.
- **macOS בלבד.** הנתיבים וההגדרות של `launchd` מניחים macOS כיום.
