# HeptaSieve

**Приватность прежде всего, безопасность для ИИ. Локальный инструмент для непрерывной синхронизации Heptabase со структурированным Markdown.**

Вы решаете, какие именно карточки ИИ-агент может видеть. Всё остальное остаётся вне его досягаемости.

## Зачем это нужно

Всё началось с простой цели: подключить Heptabase к Claude Code, чтобы ИИ-агент мог читать мои заметки.

Официальный путь: собственный [CLI](https://github.com/heptameta/heptabase-cli-skills) Heptabase, который активируется в приложении через Settings, AI Features, CLI. Он работает по принципу **fail-open**: после авторизации агент может читать всю базу знаний. Сторонние инструменты вроде сервера `heptabase-mcp` работают так же. Это нормально, если всё содержимое базы знаний можно свободно передавать. Но если конфиденциальные карточки хранятся рядом с теми, которые предназначены для ИИ, а так делает большинство пользователей, этот путь не подходит.

Главное понимание: стена приватности должна стоять на границе того, *что ИИ может прочитать*. Эта граница находится за пределами Heptabase, в том, как вы передаёте заметки агенту. Поэтому настоящая задача такого инструмента: **держать конфиденциальные карточки там, куда ИИ не добраться, и экспортировать только остальное в Markdown, доступный ИИ.** Синхронизация заметок является лёгкой половиной работы.

Это и есть sieve (сито). Проходят только те карточки, которые вы разрешили.

## Что умеет

HeptaSieve читает локальную базу данных Heptabase напрямую и записывает выбранные карточки как Markdown-файлы в указанные вами места. Задача `launchd` запускает её каждые 15 минут, чтобы Markdown оставался синхронизированным с заметками. ИИ-агент читает только папку с экспортированными Markdown-файлами. К базе данных он никогда не обращается.

- **Читает «живую» локальную базу данных.** В конце 2025 года Heptabase прекратила предоставлять [автоматические локальные резервные копии](https://support.heptabase.com/en/articles/11064116-how-does-auto-backup-work-in-heptabase), поэтому чтение активной БД стало надёжным способом непрерывной локальной синхронизации.
- **Конвертация с сохранением структуры.** Таблицы, списки bullet / todo / toggle, вложенные разделы и видео извлекаются с помощью обратной разработки схемы ProseMirror Heptabase и конвертируются в чистый Markdown.
- **Маршрутизация в любое место.** Каждая белая доска может попасть в отдельную папку, включая абсолютный путь, который помещает доску прямо в другой проект.

## Модель приватности fail-closed

Карточка экспортируется только при совпадении с одним из двух явных списков разрешений. По умолчанию ничего не читается.

| Источник | Правило |
|---|---|
| **`whitelist_whiteboards`** | Белые доски, которые вы называете. Читаются только карточки на *поверхности* каждой доски. Под-доски не отслеживаются. Чтобы включить под-доску, назовите её тоже. |
| **`card_map`** | Слой `заголовок -> точный путь`. Эти заголовки всегда синхронизируются, их путь имеет приоритет. |
| **`blacklist_whiteboards`** | Карточки с этих досок вычитаются *до* чтения любого контента. Чёрный список перекрывает белый, поэтому карточка, случайно размещённая на двух досках, всё равно будет заблокирована. |
| **Под-доски (без названия)** | Перемещение карточки на под-доску меняет её `whiteboard_id`, поэтому сканирование поверхности её не увидит. Исключено структурой, а не правилом, которое нужно помнить. |

Гарантия в одной строке: каждый запрос, затрагивающий заголовок или содержимое карточки, ограничен идентификаторами белых досок из белого списка или заголовками `card_map`. Заголовок и содержимое карточки, не входящей в белый список, никогда не загружаются в память.

Из этого следуют два принципа проектирования. **Структурное исключение лучше вычитающего исключения**: карточка, недоступная для запроса, безопаснее карточки, отфильтрованной после чтения. **Лучшее уведомление: то, которое не нужно**: инструмент спроектирован так, чтобы вам никогда не приходилось задаваться вопросом, утекла ли карточка.

## Сравнение

| | HeptaSieve | Официальный CLI Heptabase | Другие инструменты экспорта |
|---|---|---|---|
| Модель приватности | Список разрешений fail-closed | Fail-open (вся база знаний) | Полный экспорт |
| Непрерывная локальная синхронизация | Да (`launchd`, 15 мин) | Чтение по запросу | Однократный экспорт |
| Чтение «живой» локальной БД | Да | Зависит от ситуации | Часто требует файл резервной копии |
| Markdown с сохранением структуры | Таблицы, списки, разделы, видео | Зависит от ситуации | Зависит от ситуации |
| Маршрутизация по месту назначения для каждой доски | Да, включая абсолютные пути | Нет | Нет |

Это не полная замена Heptabase, и «лучше официального» справедливо только по трём параметрам: управляемая приватность, непрерывная локальная синхронизация и точность структуры. Целевая аудитория намеренно узкая: пользователи macOS, которые живут в Heptabase и заботятся о том, что видит ИИ. Если это про вас, инструмент создан именно для вашей ситуации.

## Установка

Требования: macOS, Python 3.9+, установленное десктопное приложение Heptabase.

```bash
git clone https://github.com/yyu0310/heptasieve.git
cd heptasieve
cp config.example.json config.json
```

Затем отредактируйте `config.json` (каждое поле имеет встроенный комментарий с пояснением):

1. Убедитесь, что `db_path` указывает на ваш локальный `hepta.db`.
2. Установите `base_output_dir` и `board_output_dir` в место, куда нужно записывать Markdown.
3. Перечислите белые доски для экспорта в `whitelist_whiteboards`.
4. Добавьте в `card_map` заголовки, для которых нужны точные пути.

Сначала запустите предварительный просмотр, который ничего не записывает:

```bash
python3 heptabase_sync.py --dry
```

Когда план выглядит правильно, запустите по-настоящему:

```bash
python3 heptabase_sync.py
```

### Автоматическая синхронизация каждые 15 минут

```bash
cp com.example.heptasieve.plist ~/Library/LaunchAgents/
# отредактируйте скопированный файл: укажите абсолютные пути и проверьте путь к python3
launchctl load ~/Library/LaunchAgents/com.example.heptasieve.plist
```

## Использование с ИИ-агентом

HeptaSieve поставляется с документацией, читаемой агентами, чтобы вы могли настроить её в разговоре с ИИ-агентом для кодирования вместо выполнения шагов вручную:

- [`AGENTS.md`](AGENTS.md) и [`CLAUDE.md`](CLAUDE.md): как агент должен понимать и настраивать этот инструмент.
- [`llms.txt`](llms.txt): индекс документации для языковых моделей.
- [`skills/setup-heptasieve/`](skills/setup-heptasieve/): навык Claude Code, который проводит через всю настройку за один запрос.

Направьте агента в папку с экспортированными Markdown-файлами, и никогда к `hepta.db`. Это разделение и есть суть всего инструмента.

## Как это работает

Подробности архитектуры см. в [`ARCHITECTURE.md`](ARCHITECTURE.md): поток данных, порядок fail-closed внутри `build_plan`, таблицы базы данных, которые читаются, и инварианты приватности, которые необходимо сохранять при изменении кода.

## Ограничения и честные предупреждения

- **Схема хрупкая.** Зависит от внутренней структуры базы данных Heptabase. Обновление Heptabase может её сломать. Инструмент неофициален по своей природе.
- **Чтение «живой» БД официально не одобрено.** На практике работает хорошо и является только чтением, но вы должны знать, что это не поддерживаемая интеграция.
- **Только macOS.** Пути и настройка `launchd` сейчас рассчитаны на macOS.
